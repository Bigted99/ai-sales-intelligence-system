# LeadAI Agent Instructions

Read this file before making any changes.

---

# Project Overview

LeadAI is an AI-first lead handling platform.

The purpose of LeadAI is to:

* Capture leads
* Qualify leads
* Track conversations
* Assist communication
* Escalate to humans
* Improve sales operations

LeadAI is NOT a generic messaging platform.

Communication channels are transports.

The AI workflow is the product.

---

# Before Writing Code

Always determine:

1. What problem is being solved?
2. Which entity owns the problem?
3. Does the solution align with existing architecture?
4. Is there already an established pattern?

Never introduce major architectural changes without explanation.

---

# Core Entities

Client

Represents a business.

Lead

Represents a potential customer.

Conversation

Stores communication history.

Lead Activity

Stores actions related to a lead.

Knowledge Document

Stores business knowledge.

Workflow

Stores automation logic.

Agent

Represents LeadAI intelligence systems.

---

# Architectural Rules

Lead is the primary business entity.

Conversations belong to leads.

Activities belong to leads.

Knowledge belongs to clients.

Inbox exists to support lead handling.

Do not build features that prioritize messaging over lead operations.

---

# Current Inbox Rules

Messages page:

* One row per lead.
* Show latest message preview.
* Show latest activity timestamp.
* Show unread indicators.

Conversation page:

* Shows full conversation history.
* Supports AI, Lead, and Human sender types.
* Displays messages chronologically.

---

# Sender Types

Lead

* Incoming message.
* Gray styling.

AI

* Generated by LeadAI.
* Blue styling.

Human

* Sent by user.
* Green styling.

# Coding Standards

Prefer existing patterns.

Prefer reusable components.

Prefer TypeScript types over any.

Prefer utility functions over duplicated logic.

Prefer server-side security for sensitive actions.

Avoid premature abstractions.

Keep components focused.

---

# Database Guidelines

Do not create duplicate entities.

Reuse existing relationships.

Lead remains the parent operational entity.

Conversation history should remain queryable by lead.

Document schema changes clearly.

---

# Required Workflow

Before implementation:

1. Explain the proposed solution.
2. List files that will change.
3. Identify risks.
4. Implement only after approval.

After implementation:

1. Explain changes made.
2. List affected files.
3. Note future improvements.

---

# Non Goals

Do not optimize for:

* Instagram management
* Social media scheduling
* Generic CRM features
* Generic helpdesk functionality

Optimize for:

* Lead handling
* Sales operations
* AI assistance
* Business intelligence

These principles take priority over convenience.
